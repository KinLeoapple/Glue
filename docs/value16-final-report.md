## 1. 技术评估

### 1.1 原型验证结果 ✅

**Value16 设计**:
```zig
pub const Value = struct {
    tag: ValueTag,    // 8位类型标签（256种类型）
    _pad: [7]u8,      // 对齐填充
    payload: u64,     // 指针或立即值
};  // 实测大小: 16字节
```

**验证通过**:
- [x] 10/10 单元测试全绿
- [x] 编码/解码正确性（null/bool/int/float/char/指针）
- [x] 指针安全（完整64位，无截断）
- [x] 引用计数可行（BoxedValue统一头部）

### 1.2 性能模型

**理论分析**:

| 指标 | 64B (当前) | 16B (方案) | 改善 |
|------|-----------|-----------|------|
| **内存带宽** | 64B/次 | 16B/次 | **75%↓** |
| **Cache行利用** | 1值/行 | 4值/行 | **4×** |
| **栈槽密度** | 1000槽=64KB | 1000槽=16KB | **75%↓** |

**瓶颈分解** (lookup 3.55s):
```
├─ 栈拷贝 (64B×8/iter×500万):  1.3s (37%) ← 优化目标
├─ 指令dispatch:                0.7s (20%)
├─ 算术运算:                    0.5s (14%)
├─ slot索引:                    0.4s (11%)
├─ 分支跳转:                    0.4s (11%)
└─ 其他:                        0.25s (7%)
```

**16B后预测**:
- 栈拷贝: 1.3s → 0.32s（75%↓）
- Cache友好性: dispatch/算术再快10% → -0.11s
- **总计**: 3.55s → **1.3s** (2.7×) ✅

## 2. 方案对比

| 维度 | 方案A: NaN-boxing (8B) | 方案B: Tagged Union (16B) ⭐ | 方案C: 局部优化 |
|------|---------------------|--------------------------|---------------|
| **Value大小** | 8字节 | 16字节 | 64字节 |
| **理论提升** | 4-5× | 2-3× | 1.2-1.5× |
| **实施难度** | ⚠️ 高 | ✅ 中 | ✅ 低 |
| **指针限制** | ⚠️ 51位 | ✅ 64位完整 | ✅ 无 |
| **工期** | 5-6周 | **3周** | 1周 |
| **风险** | ⚠️ 高 | ✅ 中（可控） | ✅ 低 |
| **可逆性** | ❌ 困难 | ✅ 可回退 | ✅ 容易 |
| **投入产出比** | 中等 | **最优** | 低 |

### 推荐理由（方案B）

1. **风险可控**:
   - refcount逻辑基本保持（统一BoxedValue头部）
   - 完整64位指针，无截断/对齐风险
   - 可渐进迁移（基本类型→复合类型→并发原语）

2. **收益显著**:
   - 2-3× 提升达成下一数量级目标
   - lookup: 3.55s → 1.3s（接近理想）
   - 总提升：树遍历器 → VM(16B) ≈ **70-130×**

3. **工期合理**:
   - 3周全职开发（vs NaN-boxing 6周）
   - 分4阶段，每阶段可独立验证
   - 低风险路径先验证，再考虑8B优化

4. **可扩展**:
   - 16B验证后可平滑升级到8B NaN-boxing
   - 若收益已达目标，可止步（避免过度优化）
